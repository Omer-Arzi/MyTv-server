# Secondary Metadata-Provider Strategy

**Status**: design/research document. Report only — no API calls made, no schema changes, no enrichment applied. Written to answer one question: should MyTv add a metadata provider besides TMDb for the catalog-matching cases TMDb alone can't resolve, and if so, which one(s) and how.

## 1. Why this question exists now

TMDb enrichment (`tmdb-enrichment/`) is working and applied for 184+ of 433 series. But a cluster of real, currently-unresolved cases keeps recurring across this project's own audits and manual corrections:

- **Mom** — catalog stops at Season 5 Episode 2; the user is actually at S5E13. Not an anime-numbering problem — a plain missing-episodes problem for a mainstream US network sitcom.
- **One Piece** — absolute episode #1158 (the user's next episode, likely S23E3) doesn't exist in the catalog yet; TV Time/MyTv splits this into 24 seasons, and the docs already note a documented risk of TMDb representing long-running anime with different season boundaries.
- **InuYasha / InuYasha: The Final Act** — two separate, unenriched Series rows; unclear whether "Final Act" is a continuation or should stay a distinct entry. On the explicit "do not trust" list (`docs/episode-numbering-and-season-shift-risk.md` §5).
- **JUJUTSU KAISEN** — TV Time/MyTv splits it 24+23+7=54 episodes across 3 seasons; a prior dry-run found TMDb represents it as **one absolute-numbered season of 59** — a structural disagreement, not a confidence-score problem. Also on the risk list.
- **Devil May Cry (2025)** — only one season known; `watchedEpisodeCount === knownEpisodeCount` turned out to be a false signal of "caught up" because the catalog itself may be incomplete, not because nothing more has aired.

These share a common shape: **TMDb's season/episode structure sometimes disagrees with TV Time's own structure**, and no confidence-score threshold fixes a structural disagreement — see `docs/episode-numbering-and-season-shift-risk.md` §3 for why this is explicitly called out as a different failure mode than a low-confidence match.

## 2. Current architecture (reviewed for this document)

- **`tmdb-enrichment/`** — the working, primary enrichment pipeline. `tmdb-client.ts` (Bearer-token auth, exponential backoff, no documented `Retry-After`), `scoring.ts` (title 50 / year 30 / rank-based-relevance 20, `AUTO_MATCH` ≥85 with a 15-point gap / `NEEDS_REVIEW` 50–84 / `NO_MATCH` below), `data-quality.ts` (placeholder-title, remake-collision, duplicate-title-group detectors), `single-series-safety.ts` (a separate, more permissive gate for targeted single-series fixes vs. the batch validator), `apply-plan-types.ts`/`apply-plan.ts` (dry-run-plan → reviewed → apply, never re-scores at apply time — reads only the frozen plan).
- **`secondary-provider-audit/`** — **this already IS a secondary-provider integration**, just scoped to audit/comparison rather than enrichment. `tvmaze-client.ts` (no auth at all, ~18 req/10s), `tvmaze-scoring.ts` (same 0–100 shape as TMDb's), `tvmaze-compare.ts` (`categorizeComparison`: `TMDB_LOOKS_CORRECT` / `TVMAZE_LOOKS_BETTER` / `BOTH_AGREE` / `BOTH_UNCERTAIN` / `POSSIBLE_REMAKE_COLLISION` / `POSSIBLE_ANIME_NUMBERING_MISMATCH` / `POSSIBLE_SPECIALS_MISMATCH` / `WATCHED_COUNT_EXCEEDS_PROVIDER_CATALOG` / `NO_GOOD_MATCH`). Its own code comment already documents the exact failure mode this whole document is about: *"real-world check against TVmaze's own API (Naruto: TVmaze numbers seasons by year, 2002-2007; TV Time/typical TMDb numbering uses sequential Season 1-5) shows these conventions routinely disagree."*
- **`ExternalIds` model** — `seriesId` (unique), `traktId`/`tmdbId`/`imdbId` (each unique, nullable), plus **provider-agnostic** `provider`/`providerId`/`matchConfidence`/`matchSource`/`matchedAt`/`rawMetadata`. This is a good sign: the generic fields already anticipate "some provider matched this, with some confidence, from some source" without hardcoding TMDb-specific naming — but it's a **1:1 row per series**, not built for multiple providers' opinions to coexist.
- **Enrichment apply-plan format** (`tmdb-apply-plan.json`) — a frozen, reviewed snapshot (`safeApplyCandidates`, `excludedFromCandidateUnion`, `manualReview` broken out by reason, `dangerousExamples`). The apply step is a pure "read this file, write exactly this" — no live re-matching. This same shape (plan → review → frozen apply) is the template this document reuses for a provider-override workflow (§7.4).
- **`manual-progress-corrections/`** (just built this session) — proves out the exact "plan → decisions template → dry-run → apply, with a defense-in-depth re-check at apply time" pattern this document recommends generalizing to provider selection (§7.4).
- **Trakt** (`trakt-enrichment/`) — a **complete, working dry-run pipeline already exists**, but is blocked: `docs/tmdb-enrichment-plan.md` §0 states plainly that Trakt OAuth application registration is currently broken for this account, which is *why* the TMDb pipeline was built as a parallel track in the first place.

## 4. Per-title recommendation

| Title | Best-fit provider | Why |
|---|---|---|
| **Mom** | **TVmaze** (try first — zero new integration cost), TheTVDB as fallback | Plain missing-episode-catalog problem for a mainstream US network sitcom — exactly TVmaze's historical strength, and it's already built and tested in this repo. |
| **One Piece** | **TheTVDB** (absolute-order field is the direct fix), **AniList** as a season/arc-boundary cross-check | The problem is structural (episode #1158 → S23E3 mapping), not a confidence-score problem — needs a provider that natively models absolute vs. season order. |
| **InuYasha / InuYasha: The Final Act** | **AniList** (relations graph) to confirm the sequel relationship, **TheTVDB** for absolute ordering once confirmed, **Wikidata** as an independent corroboration if needed | The open question is "are these the same continuity or not" — a relations graph answers that more directly than title/year scoring ever can. |
| **JUJUTSU KAISEN** | **TheTVDB** (absolute-order fix), **AniList** to confirm the cour/season boundaries as actually broadcast | Same class of problem as One Piece — TMDb's single absolute season of 59 vs. TV Time's 24+23+7=54 needs a provider built for exactly this distinction. |
| **Devil May Cry (2025)** | **TMDb re-check first**, **TVmaze** as a quick no-auth secondary confirmation | Recent, Western-produced (Netflix), non-anime-numbering-risk show — this is a plain "has Season 2 been added to TMDb yet" question, not a structural mismatch. |
| **Doctor Who** | **TVmaze** (meticulous BBC-content coverage, already built), **TheTVDB** as a mature alternative | Both providers have long, complete British-TV histories; TVmaze is the lower-cost first try. |
| **Avatar** | **Wikidata first** (disambiguate animated vs. live-action explicitly), then whichever catalog provider is chosen, fed the now-confirmed ID | This is a title-collision problem, not a numbering problem — solve identity first, catalog second. |

## 5. Recommended minimal architecture

1. **Keep TMDb as primary.** It's working, tested, and already applied for the majority of enriched series. Nothing here proposes replacing it.
2. **Add a secondary-provider layer used only for comparison/disambiguation, never as a silent replacement** — this generalizes the pattern `secondary-provider-audit/` already established for TVmaze. A series only gets a secondary-provider lookup when it's flagged by an existing risk signal (the episode-numbering risk list, a duplicate-title group, `POSSIBLE_ANIME_NUMBERING_MISMATCH`, or a `DATA_INCOMPLETE` manual-progress-correction finding like Devil May Cry) — not run against the whole catalog by default.
3. **Provider-specific confidence scoring, same shape as today.** `tmdb-enrichment/scoring.ts` already adapted `trakt-enrichment/scoring.ts`'s title(50)/year(30)/relevance(20) shape without sharing TMDb-specific code — a TheTVDB or AniList scorer should do the same: same weights and tier thresholds for direct comparability, provider-specific field mapping kept separate (never a shared "generic provider" abstraction that has to compromise on each API's actual shape).
4. **Store provider match decisions per series — schema direction only, not implemented here.** `ExternalIds` today is a good sign (generic `provider`/`providerId`/`matchConfidence`/`matchSource` fields already exist) but is a 1:1 row per series, which can't hold "TMDb says X with 90 confidence AND TheTVDB says Y with 75 confidence" at the same time. Two directions, to be decided in a future schema-design pass (**no schema change proposed by this document**):
   - **(a)** Add more nullable per-provider ID columns directly to `ExternalIds` (`tvdbId`, `anilistId`, `malId`, `kitsuId`, paralleling the existing `traktId`/`tmdbId`/`imdbId`) — simplest, but doesn't scale past a handful of providers before the table becomes an ever-growing list of nullable columns.
   - **(b)** A new child table (e.g. `ProviderMatch`: `seriesId`, `provider`, `providerId`, `confidence`, `matchSource`, `matchedAt`, `rawMetadata`, `decision`) allowing many rows per series. **Recommended** once more than 2–3 providers are actually in play, since it scales cleanly and mirrors the multi-row shape `manual-progress-corrections/` already uses for per-series decisions.
5. **Allow manual provider override per series, reusing the exact `manual-progress-corrections/` pattern**: a plan (per-series, per-provider candidate) → a decisions template (`apply` / `skip` / `needs_mapping` / `report_only`) → dry-run → apply, with the same defense-in-depth re-check at apply time this session's `apply-logic.ts` already does (re-verify current state hasn't drifted before writing). Never auto-decided from scores alone for a flagged title.
6. **Prevent automatic enrichment when providers disagree.** This is a direct extension of gates that already exist: `single-series-safety.ts`'s `closeCompetitorDetected`/`animeNumberingRiskDetected`/`isDataQualityFlagged` and `tvmaze-compare.ts`'s `POSSIBLE_ANIME_NUMBERING_MISMATCH`/`POSSIBLE_SPECIALS_MISMATCH` categories are exactly this concept for two providers. Generalizing to N providers: if any two confident (`AUTO_MATCH`-tier) sources propose meaningfully different episode/season structure for the same series, hard-block auto-apply and route to manual review — matching `docs/episode-numbering-and-season-shift-risk.md`'s existing explicit rule that a flagged series must never be auto-applied.

## 6. Next smallest implementation step

Two options were suggested; recommending the first as strictly smaller and lower-risk:

**Recommended: a targeted TVmaze comparison run against this session's ~10 risky titles** (Mom, One Piece, InuYasha, InuYasha: The Final Act, JUJUTSU KAISEN, Devil May Cry (2025), Doctor Who (2005), Doctor Who (2023), Avatar: The Last Airbender, Ranma ½, Hunter x Hunter) — **zero new provider-integration code**, since `secondary-provider-audit/`'s client, scoring, and comparison logic already exist and are tested. This would need only a small, parameterized run script (accept a title list instead of the full catalog) reusing `run-tvmaze-dry-run.ts`'s existing shape. Output: a concrete read on how far TVmaze alone gets on exactly the titles causing trouble today, before investing in a brand-new provider client.

**Second step, conditional on the first**: if TVmaze proves insufficient for the anime-numbering cases specifically (expected for One Piece/InuYasha/JUJUTSU KAISEN, given TVmaze's own season-by-broadcast-year convention noted in `tvmaze-compare.ts`'s header comment) — build a minimal, read-only TheTVDB client modeled directly on `tmdb-client.ts`/`tvmaze-client.ts`'s shape (same throttle/backoff pattern), scoped **only** to the confirmed anime-numbering-mismatch titles rather than the whole catalog, specifically to evaluate its absolute-order field against this project's real data before deciding whether to build it out further.

Both steps remain read-only/dry-run only, following every existing convention in this repo (no schema changes, no live writes, always a frozen plan reviewed before any apply step is even built).
